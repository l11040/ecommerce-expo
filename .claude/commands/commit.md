# 커밋 메시지 작성 스킬

변경 사항을 분석하여 컨벤션에 맞는 커밋 메시지를 한글로 작성합니다.

## 실행 지침

### 1단계: 변경 사항 확인

다음 명령어를 실행하여 스테이징된 변경 사항을 확인하세요:

```bash
# 스테이징된 변경 사항 확인
git diff --cached

# 스테이징된 파일 목록
git diff --cached --name-status
```

### 2단계: 커밋 타입 결정

| 타입 | 설명 | 예시 |
|------|------|------|
| `feat` | 새로운 기능 추가 | `feat: 사용자 로그인 기능 추가` |
| `fix` | 버그 수정 | `fix: null 포인터 에러 해결` |
| `docs` | 문서 변경 | `docs: API 문서 업데이트` |
| `style` | 코드 포맷팅 (동작 변경 없음) | `style: prettier로 코드 포맷팅` |
| `refactor` | 리팩토링 (기능 변경 없음) | `refactor: 헬퍼 함수 분리` |
| `test` | 테스트 추가/수정 | `test: 인증 유닛 테스트 추가` |
| `chore` | 빌드, 설정 등 기타 | `chore: 의존성 업데이트` |
| `perf` | 성능 개선 | `perf: 데이터베이스 쿼리 최적화` |
| `ci` | CI/CD 설정 변경 | `ci: GitHub Actions 워크플로우 추가` |

### 3단계: 커밋 메시지 형식

```
<type>(<scope>): <타이틀>

<본문>
```

#### 규칙

1. **타입**: 위 표에서 선택 (필수, 영문)
2. **스코프**: 변경 범위 (선택, 예: `auth`, `api`, `ui`)
3. **타이틀**:
   - 50자 이내
   - **한글로 작성**
   - 명사형으로 끝내기 (추가, 수정, 개선 등)
   - 마침표 없음
4. **본문**:
   - 한 줄 띄우고 작성
   - **한글로 작성**
   - 변경 이유와 내용 설명

### 4단계: 커밋 메시지 제안

변경 사항을 분석한 후:

1. 커밋 타이틀과 본문을 제안
2. 사용자 확인 요청
3. 확인 후 커밋 실행:

```bash
git add <files>
git commit -m "<title>" -m "<body>"
```

### 예시

**변경 사항**: 로그인 API에 유효성 검사 추가

```
feat(auth): 로그인 API 입력 유효성 검사 추가

- 이메일 형식 유효성 검사 추가
- 비밀번호 길이 검사 추가
- 적절한 에러 메시지 반환
```

**변경 사항**: 버튼 클릭 시 에러 수정

```
fix(ui): 버튼 클릭 이벤트 미작동 문제 해결

this 컨텍스트가 올바르게 바인딩되지 않아
이벤트 핸들러가 작동하지 않는 문제를
화살표 함수를 사용하여 수정
```

### 공용 컴포넌트 수정 시 이슈번호 포함

`src/components/` 하위의 공용 컴포넌트를 수정한 경우:

1. 해당 컴포넌트 파일 상단의 주석에서 이슈번호 확인
   - 형식: `// component-path #이슈번호`
   - 예: `// navigation/app-bar #401`
2. 커밋 메시지 본문에 이슈번호 포함
   - 형식: `[#이슈번호]`

**예시**:
```
fix(app-bar): 스크롤 숨김 시 safe-area 영역 노출 문제 해결 #401

- AppBar가 스크롤로 숨겨질 때 노치 영역을 가리는 커버 추가
- bgColor prop 추가로 배경색 통합 관리
```

### 주의사항

- 하나의 커밋에는 하나의 논리적 변경만 포함
- 변경 사항이 여러 종류면 커밋을 분리할 것을 제안
- 민감한 정보(비밀번호, API 키 등)가 포함되지 않았는지 확인
